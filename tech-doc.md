# 学校抢票小程序技术方案

## 1. 需求分析

### 核心功能
- 学校发布活动并生成抢票二维码
- 学生通过公众号扫码进入小程序抢票
- 采用先到先得机制
- 成功抢票后生成带有专属编号和座位号的纪念票根
- 支持几万人并发抢票

### 非功能需求
- 高并发处理能力
- 系统稳定性
- 数据一致性
- 响应速度快

## 2. 技术栈选择

### 后端
- **语言**：Node.js (TypeScript)
- **框架**：Nest.js - 模块化、可扩展、支持TypeScript
- **数据库**：PostgreSQL - 支持事务和并发控制
- **缓存**：Redis - 用于缓存活动信息和限流
- **消息队列**：RabbitMQ - 用于异步处理抢票请求
- **认证**：微信小程序登录

### 前端
- **框架**：微信小程序原生开发
- **UI组件**：微信小程序原生组件
- **状态管理**：小程序自带数据绑定

## 3. 系统架构

### 架构图
```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│ 微信小程序前端   │────>│  Nest.js 后端    │────>│  PostgreSQL 数据库 │
└─────────────────┘     └─────────────────┘     └─────────────────┘
                              ↑                      ↑
                              │                      │
                        ┌─────────────────┐     ┌─────────────────┐
                        │    Redis 缓存   │<────│  RabbitMQ 消息队列 │
                        └─────────────────┘     └─────────────────┘
```

### 核心流程
1. 管理员创建活动并设置票数
2. 系统生成活动二维码
3. 学生扫码进入小程序
4. 学生点击抢票按钮
5. 后端处理抢票请求（并发控制）
6. 成功抢票后生成票根
7. 前端展示票根

## 4. 并发处理机制

### 1. 数据库层面
- 使用 PostgreSQL 事务确保数据一致性
- 采用乐观锁（版本号）避免并发冲突
- 使用 `SELECT FOR UPDATE` 锁定行记录

### 2. 缓存层面
- Redis 缓存活动剩余票数
- 使用 Redis 分布式锁防止重复抢票
- 设置 Redis 过期时间确保数据一致性

### 3. 消息队列层面
- 使用 RabbitMQ 异步处理抢票请求
- 限流控制，避免系统过载
- 确保每个请求都能被处理

### 4. 应用层面
- 实现令牌桶算法进行限流
- 对同一用户的请求进行频率限制
- 服务降级策略，确保系统稳定

## 5. 性能优化

- 数据库索引优化
- Redis 缓存策略
- 代码层面的优化（如使用 async/await、避免同步操作）
- 负载均衡

## 6. 安全考虑

- 防止 SQL 注入
- 防止 XSS 攻击
- 防止 CSRF 攻击
- 微信小程序登录认证
- 数据加密存储

## 7. 部署方案

- 使用 Docker 容器化部署
- 配置负载均衡
- 监控系统（如 Prometheus + Grafana）
- 日志系统（如 ELK 栈）
