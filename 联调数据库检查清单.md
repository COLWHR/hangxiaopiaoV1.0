# 联调数据库检查清单

这份清单用于整合版项目联调时边操作边核对数据库变化。

适用仓库：
- `E:\Study\codes\hangxiaopiao_Test`

当前数据库类型：
- `SQLite`

当前数据库文件：
- `backend/database.db`

建议使用方式：
1. 先启动后端，确保 `database.db` 已生成。
2. 用 Navicat 连接 `backend/database.db`。
3. 每做完一个页面动作，就执行当前动作对应的 SQL。
4. 只记录“哪些字段变了、是否符合预期”，不要一上来把所有表都翻一遍。

---

## 一、联调前基线检查

先执行这几条，记住当前总量，后面方便判断有没有新增或更新。

### 1. 管理员账号

```sql
SELECT id, account, displayName, adminAccountId, adminUserId
FROM admin_accounts;
```

预期：
- 至少有一条默认管理员
- 常用测试账号通常是 `ADMIN2024`

### 2. 活动列表

```sql
SELECT id, title, status, totalTickets, availableTickets, startTime, endTime, createdAt, updatedAt
FROM activities
ORDER BY id DESC;
```

预期：
- 能看到已有活动数量
- 记住你要测试的活动 ID

### 3. 用户列表

```sql
SELECT id, phone, name, studentId, profileCompleted, createdAt, updatedAt
FROM users
ORDER BY id DESC;
```

预期：
- 知道当前测试手机号是否已存在

### 4. 票据列表

```sql
SELECT id, activityId, userId, ticketNumber, status, createdAt
FROM tickets
ORDER BY id DESC;
```

预期：
- 记住当前票据总数

## 二、管理员端联调

## 1. 管理员登录

动作：
- 管理员端输入账号密码登录

执行 SQL：

```sql
SELECT id, account, displayName, adminAccountId, adminUserId
FROM admin_accounts
WHERE account = 'ADMIN2024';
```

重点看：
- 这条管理员记录是否存在
- `adminAccountId`
- `adminUserId`

预期：
- 登录动作本身一般不改数据库
- 这一步主要是确认后面保存草稿、发布活动要用到的管理员标识是存在的

---

## 2. 保存草稿

动作：
- 在发布页填写部分或全部活动信息
- 点击“保存草稿”

执行 SQL：

```sql
SELECT id, adminAccountId, adminUserId, draftData, lastClearedAt, createdAt, updatedAt
FROM admin_activity_drafts
ORDER BY id DESC;
```

重点看：
- 是否新增了一条草稿
- 或原草稿的 `updatedAt` 是否更新
- `draftData` 里是否包含你刚填写的标题、地点、票数、时间等内容
- `lastClearedAt` 通常应为 `NULL`

预期：
- 同一个管理员反复保存草稿，通常是更新同一条记录

异常判断：
- 没新增也没更新时间变化：前端可能没真正请求到后端
- `draftData` 空着：后端收到了请求，但保存内容可能有映射问题

---

## 3. 预览活动

动作：
- 点击“预览”

执行 SQL 1：

```sql
SELECT id, adminAccountId, adminUserId, updatedAt
FROM admin_activity_drafts
ORDER BY id DESC;
```

执行 SQL 2：

```sql
SELECT id, title, status, createdAt, updatedAt
FROM activities
ORDER BY id DESC;
```

重点看：
- 预览是否只是前端展示
- 是否没有新建正式活动
- 是否只读取草稿而不改表

预期：
- 多数情况下，预览动作本身不应该新建 `activities` 记录

异常判断：
- 只点预览就新增了正式活动：说明预览和发布逻辑混在一起了

---

## 4. 发布活动

动作：
- 在发布页点击“发布活动”

执行 SQL 1：

```sql
SELECT id, title, description, location, totalTickets, availableTickets, startTime, endTime, status, adminAccountId, adminUserId, qrCodeUrl, createdAt, updatedAt
FROM activities
ORDER BY id DESC;
```

执行 SQL 2：

```sql
SELECT id, adminAccountId, adminUserId, draftData, lastClearedAt, updatedAt
FROM admin_activity_drafts
ORDER BY id DESC;
```

重点看：
- `activities` 是否新增一条正式活动
- 标题、时间、地点、票数是否和页面填写一致
- `totalTickets` 是否正确
- `availableTickets` 初始是否等于 `totalTickets`
- `status` 是否为 `published` 或系统当前实现的发布状态
- `adminAccountId` 和 `adminUserId` 是否正确
- `qrCodeUrl` 是否生成

草稿补充看：
- 发布后草稿是否保留
- 或 `draftData` 是否被清空
- 或 `lastClearedAt` 是否被写入

预期：
- 发布后一定要能在 `activities` 看到正式记录

异常判断：
- 页面提示发布成功，但 `activities` 没新增：大概率是假成功或前端只改了本地状态

---

## 5. 修改活动

动作：
- 在管理员端修改已发布活动

执行 SQL：

```sql
SELECT id, title, description, location, totalTickets, availableTickets, startTime, endTime, status, updatedAt
FROM activities
WHERE id = 你的活动ID;
```

重点看：
- 你修改的字段是否真的更新
- `updatedAt` 是否刷新
- 如果修改了票数，`totalTickets` 和 `availableTickets` 是否仍然合理

预期：
- 活动记录被更新，不应重复新增一条相同活动

异常判断：
- 改完后反而新增活动：说明编辑接口和创建接口混用了
- 已有人抢票时，`availableTickets` 被错误重置成 `totalTickets`：说明票数回写逻辑有问题

---

## 三、用户端联调

## 6. 用户首次登录

动作：
- 用手机号和密码登录用户端

执行 SQL：

```sql
SELECT id, nickname, name, studentId, college, className, phone, passwordHash, profileCompleted, createdAt, updatedAt
FROM users
WHERE phone = '你的测试手机号';
```

重点看：
- 这条用户是否被创建
- `passwordHash` 是否已生成
- `profileCompleted` 是否为 `0`

预期：
- 如果这是新手机号，首次登录后通常会自动创建用户

异常判断：
- 查不到用户：请求可能没打到后端
- 用户有了但 `passwordHash` 还是空：登录流程没完整落库

---

## 7. 用户补全资料

动作：
- 在用户端填写姓名、学号、学院、班级等资料并保存

执行 SQL：

```sql
SELECT id, name, studentId, college, className, phone, profileCompleted, updatedAt
FROM users
WHERE phone = '你的测试手机号';
```

重点看：
- `name`
- `studentId`
- `college`
- `className`
- `profileCompleted`
- `updatedAt`

预期：
- 资料字段被写入
- `profileCompleted` 从 `0` 变成 `1`

异常判断：
- 页面显示保存成功，但字段没变化：前后端字段映射可能不一致

---

## 8. 用户查看活动列表或详情

动作：
- 打开活动列表
- 进入活动详情

执行 SQL：

```sql
SELECT id, title, status, totalTickets, availableTickets, startTime, endTime
FROM activities
ORDER BY id DESC;
```

重点看：
- 活动是否真实存在
- `status` 是否是前端可展示状态
- 时间是否处于合理区间

预期：
- 查看动作一般不改数据库

异常判断：
- 数据库里有活动，前端却看不到：重点查前端过滤条件、接口映射、状态判断

---

## 四、抢票联调

## 9. 抢票前基线

动作：
- 用户准备点击抢票前，先记住当前活动状态

执行 SQL：

```sql
SELECT id, title, status, totalTickets, availableTickets, startTime, endTime
FROM activities
WHERE id = 你的活动ID;
```

重点看：
- `status` 是否为 `published` 或 `active`
- `availableTickets` 是否大于 `0`
- 当前时间是否在 `startTime` 和 `endTime` 之间

预期：
- 满足以上条件后，抢票才有机会成功

常见失败原因：
- 当前时间早于 `startTime`
- 当前时间晚于 `endTime`
- `status` 不是 `published` 或 `active`
- `availableTickets = 0`

---

## 10. 抢票成功后

动作：
- 点击抢票，并且页面提示成功

执行 SQL 1：

```sql
SELECT id, activityId, userId, ticketNumber, seatNumber, status, createdAt
FROM tickets
ORDER BY id DESC;
```

执行 SQL 2：

```sql
SELECT id, ticketId, qrCodeUrl, generatedAt
FROM ticket_stubs
ORDER BY id DESC;
```

执行 SQL 3：

```sql
SELECT id, title, totalTickets, availableTickets
FROM activities
WHERE id = 你的活动ID;
```

重点看：
- `tickets` 是否新增 1 条
- `ticket_stubs` 是否新增 1 条
- `activities.availableTickets` 是否减 1

预期：
- 这 3 个变化应该同时成立

正确示例：
- 原来 `availableTickets = 200`
- 抢票成功后 `availableTickets = 199`
- `tickets` 新增 1 条
- `ticket_stubs` 新增 1 条

异常判断：
- 只有 `tickets` 新增，`availableTickets` 不变：票数扣减逻辑有问题
- `availableTickets` 减了，但没有 `tickets`：事务一致性可能有问题
- `tickets` 有了，但没有 `ticket_stubs`：票根生成逻辑有问题

---

## 11. 重复抢票

动作：
- 同一个用户对同一个活动再次点击抢票

执行 SQL：

```sql
SELECT id, activityId, userId, ticketNumber, status
FROM tickets
WHERE activityId = 你的活动ID
  AND userId = 你的用户ID;
```

重点看：
- 是否仍然只有 1 条票据

预期：
- 同一用户对同一活动只能有 1 张票

异常判断：
- 出现 2 条及以上：防重逻辑失效

## 七、常用快速排查语句

### 1. 查最新活动

```sql
SELECT id, title, status, totalTickets, availableTickets, createdAt, updatedAt
FROM activities
ORDER BY id DESC
LIMIT 5;
```

### 2. 查指定手机号用户

```sql
SELECT id, phone, name, studentId, profileCompleted, createdAt, updatedAt
FROM users
WHERE phone = '你的测试手机号';
```

### 3. 查指定活动的票据

```sql
SELECT id, activityId, userId, ticketNumber, status, createdAt
FROM tickets
WHERE activityId = 你的活动ID
ORDER BY id DESC;
```

### 4. 查指定用户的票据

```sql
SELECT id, activityId, userId, ticketNumber, status, createdAt
FROM tickets
WHERE userId = 你的用户ID
ORDER BY id DESC;
```

## 八、测试结论怎么下

你每一步测完后，尽量只下这 3 类结论之一：

1. 页面动作成功，数据库变化符合预期
2. 页面动作成功，但数据库变化不符合预期
3. 页面动作失败，数据库无变化或变化异常

这样你最后回头整理问题时会非常清楚，不容易乱。
